Work Ontology

The seven-level hierarchy that replaces tickets: Signal → Intent → Spec → Contract → Capability → Feature → Product. With three governance dimensions and an Agile comparison.

The Hierarchy

Signal ↓ Intent ↓ Spec ↓ Contract ↓ Capability ↓ Feature ↓ Product

Level by Level

1. Signal
Observations from work, research, conversation. Raw input. Includes confidence score and source. Multiple signals may relate to the same intent.
2. Intent
What we want to change. Declarative: "Users should be able to X." Connects signals to specs. One intent may spawn multiple specs.
3. Spec
How we'll change it. Narrative + acceptance criteria. Humans write specs; agents use them as contracts.
4. Contract
Verifiable assertions. The spec's assertions extracted and made executable. Done when contract passes.
5. Capability
Building blocks. Reusable functions, components, patterns that contracts are composed of.
6. Feature
User-facing unit. One or more capabilities working together. Released to production.
7. Product
System. Multiple features forming a coherent offering.

Governance Dimensions

How work is organized across three axes

Time Dimension
Continuous flow. No sprints. Work moves when dependencies resolve, not on calendar boundaries.
Quality Dimension
Contracts are the source of truth. Code and design conform to contracts, not the reverse.
Visibility Dimension
OpenTelemetry-compatible events at every transition. Work is observable at all times.

Versus Agile

How Intent differs from Agile's model

Agile: User Story → Sprint → Standup → Backlog Refinement → Retro
Ceremony-driven. Humans coordinate via ritual. Estimation guesses duration.
Intent: Signal → Intent → Spec → Contract → Observe → Event Stream
Flow-driven. Systems coordinate via dependency graphs. Observation reveals actual patterns.
