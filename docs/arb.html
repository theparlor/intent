Current Tech Stack

Everything Intent runs on today. Deliberately simple: files on disk, tracked in git, observable through events. No databases, no SaaS, no lock-in.

MCP Server
Python 3 + FastMCP
Pydantic v2 validation. Stdio transport. 7 tools exposed. Located at tools/intent-mcp/server.py
CLI Tools
Bash (POSIX strict mode)
4 executables: intent-signal, intent-intent, intent-spec, intent-status. set -euo pipefail.
Data Format
Markdown + YAML Frontmatter
All work artifacts (signals, intents, specs, contracts) are .md files with structured YAML headers.
Event System
JSONL (OTel-compatible)
15 event types, 6 emission mechanisms. Append-only file at .intent/events/events.jsonl
CI/CD
GitHub Actions
Event emission on push. Detects signal/intent/spec/decision changes via git diff.
Storage
Git + Filesystem
Zero external dependencies. Everything tracked in .intent/ directory. Portable across repos.
Documentation
Static HTML + GitHub Pages
12 pages at docs/. Inline CSS with shared custom properties. No build step.
Observability
Entire.io (git layer)
Agent reasoning traces. Enabled across 4 repos. Bridges to Intent event stream (planned).
AI Surfaces
Claude Code + Cowork + Cursor
MCP server connects to all three. Signal capture, intent proposal, spec creation, status queries.
Versioning
YYYY.MM.DD-MAJOR.MINOR.PATCH
Timestamp prefix + semver suffix. Currently at 0.7.0.

Architectural Constraints

Constraint
Git-native
Everything must live in the repo. No external databases. Portable via git clone.
Constraint
Single-developer topology
Brien is the only active builder. Tools must work for a team of one with AI augmentation.
Constraint
No build step for docs
HTML pages are hand-authored, no SSG. This keeps the docs portable but limits scale.
Constraint
Agent-executable specs
Specs must be concrete enough that Claude Code can implement without human translation.

Architectural Review Board

Five review perspectives that every roadmap item passes through before becoming a spec. The first four are Intent's hero personas. The fifth is the Claude Code execution lens — the "can I actually build this?" gut check.

△
Practitioner-Architect
Designs systems, captures signals while building
  • Does this fit the existing repo pattern (.intent/ structure)?
  • What are the integration points with current CLI/MCP tools?
  • Does this introduce a new abstraction, or extend an existing one?
  • Can I capture signals about this while I build it?
  • What's the blast radius if this goes wrong?
◇
Product-Minded Leader
Reviews clusters, prioritizes intents, investment decisions
  • Does this solve a validated problem, or are we building speculatively?
  • Which signals or clusters drove this item?
  • What's the adoption impact? Will practitioners use this?
  • Is this the constraint in the value stream right now?
  • What do we learn if this fails?
○
Design-Quality Advocate
Ensures specs meet quality bar, monitors outcomes
  • Does this create technical debt we'll pay for later?
  • Is the schema clean, or are we duct-taping?
  • How do we test this? What's the acceptance criteria?
  • Does it degrade the developer experience?
  • Is there a simpler way that achieves 80% of the value?
◉
AI Agent
Implements against specs, needs clear contracts
  • Are the acceptance criteria binary (pass/fail)?
  • Are contracts explicit enough to verify programmatically?
  • What files do I need to read to understand the context?
  • Is the scope contained to one session, or does it span multiple?
  • What existing patterns should I follow?
⚡
Claude Code Execution Lens
Feasibility, scope, dependencies, risk
  • Can this be built with Bash/Python and the current stack?
  • Is this one Claude Code session (~30 min) or does it need multiple?
  • What needs to exist in the repo before this can start?
  • What's the test strategy? Can I verify my own output?
  • What's the SIG-014 risk? (large file push, context drift potential)
