Repo Pattern

The three-layer structure every Intent-native repo follows: .intent/, .claude/, .entire/ — with file schemas and directory conventions.

Directory Structure

.intent/ signals/ intents/ specs/ contracts/ decisions/ events/ .claude/ projects/ sessions/ .entire/ traces/ src/ docs/

Layer 1: Intent (.intent/)

The Intent layer stores all work artifacts in structured files, versioned in git.

signals/

Observations from work, research, conversation. Each signal: source, confidence, timestamp, related intents. YAML or JSON files named by signal ID.

intents/

What we want to change. Declarative: "Users should be able to X." Links signals to specs. YAML with title, description, related signals, assigned to.

specs/

How we'll change it. Narrative + acceptance criteria (contracts). Markdown with frontmatter. Human-readable but machine-parseable.

contracts/

Verifiable assertions extracted from specs. JSON schema defining inputs, outputs, assertions. Executable by agents.

decisions/

Significant architectural and strategic decisions. Context, alternatives considered, rationale, decision date. Markdown with frontmatter.

events/

Structured events from execution. OpenTelemetry format. signal.created, spec.written, contract.completed, etc. JSON lines format.

Layer 2: Claude (.claude/)

Agent reasoning and session artifacts. Where Claude Code stores project context and reasoning traces.

projects/

High-level project metadata. Vision, architecture, key decisions, dependencies. Used by agents to bootstrap context.

sessions/

Session transcripts and reasoning. Agents log their thinking, decisions made, blockers encountered. Enables observability of agent cognition.

Layer 3: Entire (.entire/)

Observability layer. End-to-end execution traces and system health metrics.

traces/

Complete execution traces. Spans, logs, metrics. Ingested from GitHub Actions, agent sessions, test runs. Used by observe phase for learning.

File Naming Conventions

Signals: signal-YYYYMMDD-slug.yml

Intents: intent-YYYYMMDD-slug.yml

Specs: spec-YYYYMMDD-slug.md

Contracts: contract-spec-slug.json

Decisions: decision-YYYYMMDD-slug.md

Events: events-YYYYMMDD.jsonl
