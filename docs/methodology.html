Intent Methodology

How teams organize work around AI-augmented execution. A framework for clarity, autonomy, and continuous learning.

The Loop

Intent defines a continuous four-phase cycle that replaces time-boxed sprints:

  1. Notice: Signals emerge from customer research, agent traces, metrics, and conversations. The team surfaces insights continuously.
  2. Spec: Signals crystallize into Intent (hypothesis + outcome) and Spec (declarative behavior). This is the "why" and "what" layer.
  3. Execute: Agents and humans build against the spec. Work is observable—every decision is traced.
  4. Observe: Traces are analyzed for signal: decisions made, risks surfaced, learning captured. Observations flow back into Notice.

Three Governance Dimensions

The loop addresses the flow of work. Three orthogonal dimensions govern what gets built:

Right Things (Discovery)

Validate ideas via continuous discovery. Use opportunity trees (Teresa Torres) to surface, validate, and prioritize problems. Specs reference their parent Intent, tying each piece of work to a validated opportunity.

Right Time (Sequencing)

Parallelize and sequence work via dependency graphs, not sprint commitments. The spec layer declares dependencies explicitly. The scheduler respects them. No blocked teams.

Right Way (Architecture)

Enforce technical governance via tech radar and permit levels. Each spec cites tech radar constraints. "Play" tech requires no approval. "Build" tech requires architecture review. "Operate" tech is legacy and requires migration plans.

Work Ontology

The atomic unit shifts from "unit of human labor" to "declarative state description agents can execute against."

Intent defines a formal hierarchy of work units, each with its own schema, state machine, and contract:

  • Signal: Observation from work or research proposing a change
  • Intent: Hypothesis-driven outcome with success criteria
  • Spec: Declarative behavior description agents can execute
  • Contract: Verifiable interface agreement; done when assertions pass
  • Capability: Reusable building block; composable into features
  • Feature: User-visible value delivery; composed of capabilities
  • Product: Living system of features with continuous evolution

Observability: Entire.io

Execution is captured using Entire.io, a distributed tracing system for multi-agent work. Every agent session produces:

  • Session transcript: Full reasoning log linked to commit SHA
  • Span hierarchy: Intent → Spec → Contract mapping to OTel traces
  • Span events: Decisions, risks, questions captured as structured data

The observe cycle reads these traces and extracts signals automatically, closing the loop.

Repo Structure

Intent-native repositories mirror the loop as first-class directories:

  • .intent/ – Loop metadata (INTENT.md, decisions.md, risks.md)
  • .claude/ – Agent operating context (CLAUDE.md, commands/)
  • .entire/ – Observability layer (Entire.io traces, metadata)
