# SPEC-003 — SIG-022 ULID Migration

## Problem Statement

Per SIG-022 (captured 2026-03-30) and INT-009 P0 #1 (2026-04-09), the Intent
system used sequential ID generation for signals, intents, specs, and
contracts. IDs like `SIG-042`, `INT-007`, `SPEC-003` were generated by
scanning existing files and incrementing the max value.

This approach has three structural failure modes:

1. **Concurrent writers collide.** Two agents generating a signal simultaneously
   will both read `next_id=016` and both write `SIG-016`. No coordination
   mechanism exists.

2. **Multi-surface drift.** The local `.intent/signals/` directory, the live
   signals dashboard, and the MCP server's in-memory store all maintained
   independent counters. The 2026-03-30 Cowork session proved this — new
   signals created locally (SIG-006 through SIG-011) collided with IDs
   already in use on the dashboard.

3. **Cross-team ambiguity.** `SIG-042` from Team A means nothing to Team B
   because each team has its own sequence. Cross-engagement signal
   propagation is impossible.

SIG-022 explicitly listed this as blocking for SIG-016 (multi-agent MCP
architecture) at Phase 3+. INT-009 P0 #1 lists it as blocking for "any
second engagement coming online."

## Solution Description

Replace sequential ID generation with Crockford base32 ULIDs. Format:
`{PREFIX}-{ULID}` where ULID is 26 characters split as:
- **10 chars:** 48-bit millisecond timestamp, big-endian
- **16 chars:** 80-bit randomness from `os.urandom`

Example IDs:
- `SIG-01KNSVCCFA8YXWR6VMZG48N3ZR` (was `SIG-042`)
- `INT-01KNSVDZHJ8JHTDRH3JTYF5JQF` (was `INT-007`)

**Properties:**
- Globally unique without coordination (80 bits of entropy per ms)
- Timestamp-sortable lexicographically (ULIDs sort in creation order)
- Human-readable enough for conversation ("that SIG from early April")
- Grep-friendly (the prefix is preserved — `SIG-` still matches)
- Works for all entity types: SIG, INT, SPEC, CON, DEC, and any future prefix

**Backward compatibility:**
Existing `SIG-NNN`, `INT-NNN`, `SPEC-NNN` IDs remain valid. The match
regex accepts both formats: `SIG-([0-9]{3}|[0-9A-Z]{26})`. No migration
of existing files is required or performed.

## Implementation

Two generator libraries, one for each language used in the Intent codebase:

### `bin/lib/id_gen.sh` — shell helper

Pure bash with python3 subshell for timing + base32 encoding. Exports:
- `generate_ulid()` — returns raw 26-char ULID
- `generate_id PREFIX` — returns `PREFIX-{ulid}`
- `match_id_regex PREFIX` — returns regex pattern matching both formats
- `extract_id_from_file FILE [PREFIX]` — extract ID from frontmatter
- `is_valid_id ID PREFIX` — validate ID format
- `sort_ids` — sort stdin of IDs chronologically (legacy first, ULID second)

Shell CLIs (`bin/intent-signal`, `bin/intent-intent`, `bin/intent-spec`)
source this library and call `generate_id SIG/INT/SPEC` in place of
the legacy `next_signal_id()` / NEXT_ID loop patterns.

### `servers/id_gen.py` — Python module

Stdlib-only (no external dependencies — no `ulid-py` package needed).
Exports:
- `generate_ulid() -> str` — returns raw 26-char ULID
- `generate_id(prefix: str) -> str` — returns `PREFIX-{ulid}`
- `MATCH_ID_PATTERN` — compiled regex matching both formats
- `is_valid_id(id_str, prefix=None) -> bool`
- `parse_id(id_str) -> tuple[str, str] | None`
- `is_legacy_id(id_str) -> bool`
- `sort_ids(ids) -> list[str]` — legacy before ULID, chronological within each

Module has a `__main__` self-test block that can be run directly:
`python3 servers/id_gen.py` — exits non-zero on any failure.

## Out of Scope

- **Migration of existing files.** Legacy `SIG-NNN` files keep their IDs.
  No rename, no re-issue. The migration is forward-only.
- **Dashboard regex updates.** The signals.html dashboard lives in the
  intent-site repo and has its own regex patterns. Updating those is a
  separate follow-up (not blocking).
- **ID shortening.** ULIDs are 26 characters. Some call this ugly. No
  truncation in v1.0 — shortening breaks collision guarantees.
- **Additional entity types.** CON-NNN and DEC-NNN are supported via the
  same helpers but not actively tested in this ship.

## Test Scenarios

1. Generate 1000 ULIDs in rapid succession → zero collisions
2. Generate 100 ULIDs from two shell processes simultaneously → zero collisions
3. Mix legacy + new ID formats in a sort → legacy first, chronological
4. `grep` with the backward-compatible regex finds both formats
5. `parse_id` on malformed input returns None (doesn't raise)

All scenarios passed in the 2026-04-09 smoke test.
