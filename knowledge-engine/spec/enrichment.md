# Retroactive Enrichment

> **Purpose:** How new growth in the Knowledge Engine or new domain knowledge gets reapplied to prior work — pressure-testing existing understanding and revealing insights for newer engagements with less context.
>
> **Version:** 1.0 — 2026-04-06

---

## The Pattern

When Brien develops a new capability (e.g., autogenerating company dossiers, expanding to industry-level analysis), that capability should be **reapplied against all prior engagements** to:

1. **Pressure test** the new capability against known outcomes
2. **Enrich** prior engagement knowledge with the new lens
3. **Surface insights** that help forge forward in newer engagements
4. **Validate or invalidate** existing personas, journeys, DDRs against the new capability's output

This is the **recompilation cascade**: same raw sources, upgraded compiler, potentially different (better) output.

---

## How It Works

### Trigger

A recompilation cascade is triggered when:
- A new artifact type or operation is added to the Knowledge Engine
- A significant new capability (company dossier, industry scan, etc.) is built
- A new theme or design rationale reaches confidence ≥ 0.7 in Core
- Brien explicitly requests: "reapply this against prior engagements"

### Process

```
New capability developed
  │
  ├── 1. Apply to CORE knowledge base first
  │     → Does it reveal gaps in Core archetypes?
  │     → Does it contradict existing Core themes?
  │     → Update Core confidence scores
  │
  ├── 2. Apply to each ACTIVE engagement (most recent first)
  │     → Run against engagement's raw/ corpus
  │     → Generate new or updated engagement knowledge artifacts
  │     → Flag where engagement reality contradicts Core assumptions
  │     → Surface cross-engagement patterns not visible before
  │
  ├── 3. Apply to HISTORICAL engagements (optional, on demand)
  │     → Lower priority — historical data may be stale
  │     → But valuable for pattern recognition across career
  │     → Especially useful for new engagements in similar industries
  │
  └── 4. Promote generalizable findings to Core
        → Sanitize per federation rules
        → New themes, archetypes, or methodology improvements
```

### Example: Company Dossier Expansion

Brien builds an autogenerated company dossier capability (Skills Engine). The recompilation cascade:

1. **Apply to Subaru:** Generate a full SOA dossier. Cross-reference against existing Subaru engagement knowledge. Does the dossier reveal industry patterns Brien's been coaching around? Does it surface competitive pressures that explain why SOA is investing in transformation?

2. **Apply to ASA:** Generate an ASA dossier. Different industry (healthcare associations vs automotive). Does the product operating model playbook Brien helped build align with what the dossier reveals about ASA's strategic position?

3. **Apply to F&G:** Generate an F&G dossier. Financial services. The accelerator plan Brien proposed — does the dossier context validate or challenge the approach?

4. **Cross-engagement synthesis:** With dossiers for all three, Brien can now see patterns: "All three clients are mid-transformation with internal resistance. The consulting approach that worked at Subaru applies to F&G but NOT to ASA because..." This is the insight that was invisible without the recompilation.

5. **Promote to Core:** The cross-engagement pattern becomes a new Core theme: "Mid-transformation organizations share X characteristics. The consulting playbook needs to account for Y."

## Recompilation Scope Levels

| Level | What's Recompiled | When |
|-------|-------------------|------|
| **Artifact** | A single knowledge artifact recompiled with new evidence | On every ingest (already handled) |
| **Engagement** | All knowledge artifacts in one engagement recompiled with a new capability | When a new capability is built |
| **Cross-engagement** | Same capability applied across multiple engagements, then synthesized | When Brien wants cross-cutting insights |
| **Full cascade** | Every engagement + Core recompiled with a new lens | Major capability additions (rare) |

---

## Confidence Dynamics During Recompilation

When prior knowledge is recompiled:
- **Confirmed patterns:** Confidence increases (corroborated by recompilation)
- **Contradicted patterns:** Confidence decreases. DDR status may change to `invalidated`. Persona assumptions flagged for revision.
- **New patterns:** New artifacts created with `origin: synthetic` (generated from recompilation, not new raw sources)
- **Coverage gaps revealed:** Lint findings captured as signals

## The Career Flywheel

Brien's Knowledge Farm has 30 years of progressive experience across dozens of engagements. Each new capability developed today can be retroactively applied to that history, extracting patterns that were invisible at the time.

```
New capability
  → applied to current engagement (immediate value)
  → applied to prior engagements (pattern extraction)
  → synthesized across all engagements (career-level insights)
  → promotes to Core (methodology improvement)
  → Core improvement applied to NEXT engagement (compounding value)
```

This is the flywheel: every engagement makes Brien's Core knowledge more powerful, which makes every future engagement start from a higher baseline, which generates more knowledge to promote, which makes Core even stronger.

The Knowledge Farm is Brien's first brain. The recompilation cascade is how it stays current — not by Brien manually revisiting old work, but by the system automatically reapplying new lenses to accumulated history.
