## The thesis: differentiation is the spine + altitude, not the engine

```
        ┌─────────── the product (defensible) ───────────┐
        │   SPINE:  Notice → Spec → Execute → Observe      │
        │   ONTOLOGY: Signal→Intent→Spec→Contract→…→Product│
        │   GOVERNANCE: L0–L4 trust gates · DDRs           │
        │   CONTRACT: verbatim passthrough (S3)            │
        └───────────────────────┬─────────────────────────┘
                                 │ rests on
                    ┌────────────▼────────────┐
                    │  memory/synthesis engine │  ← commoditizing (S1), swappable
                    └──────────────────────────┘
```

**Build-allocation consequence.** Do not spend build budget making a marginally better memory
engine — that race is being run to zero by ≥4 teams. Spend it on the surfaces the category does
not reach:
1. **The spine's interface quality** — the `intent-knowledge` envelope (routing, typed supply
   policy, governed planes). This is the SPEC-DRAFT (D1–D4).
2. **The verbatim-passthrough contract** (S3) — a refusal, cheap to hold, impossible to copy.
3. **The audit/observe loop** (D4 `audit_chain`) — gap detection over the graph, which a memory
   layer has no altitude to perform.

This is consistent with the standing posture in
[[feedback_build_scope_vs_run_scope]] / [[feedback_overbuilding_calculus]]: gate the *engine*
build (low value — commoditizing), invest the *spine* build (high value — defensible).

(note/file/formula-computed properties) plus a **declarative query/view layer**

*independently*:

  unverified Contracts, orphans" is a declarative query over typed graph properties.
- **D2 supply-policy typing** is a typed-property substrate over the envelope.

Next move (folded into the SPEC-DRAFT, not executed): add computed-property + summary **"views"**
over the farm — coverage, unverified-Contract count, staleness — feeding the D4 drift color
signal. Record as: **our envelope + gap-audit are independently converging on the `.base` shape.**

The Bases skill-authoring pattern (single-purpose, composable, self-validating, spec-conformant,
portable across Claude Code / Codex / OpenCode) is also the model for shipping **Lore / Cast /
Voices as portable skills alongside the MCP surface** — the two-surface distribution shape Forge
